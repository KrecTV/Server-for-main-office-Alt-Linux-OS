Сервер головного офиса (HQ-SRV) – DNS и безопасный SSH

Пока маршрутизатор гоняет пакеты, самое время вдохнуть в сеть интеллект. Сервер HQ-SRV — это не просто ещё одна виртуалка, а главный хранитель имён и первый бастион удалённого доступа. Без него ваши внутренние машинки будут общаться по IP, а SSH будет ломиться на стандартный 22-й порт, собирая толпы ботов-сканеров. Исправляем.

Наш сервер будет сидеть в VLAN 100, обслуживать DNS-зону example.com через лёгкий dnsmasq и встречать администраторов на нестандартном порту SSH. Вперёд.

hostnamectl hostname srv-hq.example.com

exec bash

Классическое начало: даём осмысленное имя и перезапускаем оболочку, чтобы оно сразу закрепилось. Теперь приглашение командной строки будет напоминать, что мы именно на сервере, а не на роутере.

Сервер, как и было задумано, подключён к сети головного офиса через транковый порт. Значит, на физическом интерфейсе нужно поднять тегированный подынтерфейс для VLAN 100. Создаём привычную структуру конфигов ALT Linux.

cd /etc/net/ifaces

mkdir enp0s3 enp0s3.100

Физический порт enp0s3 просто объявляем Ethernet-интерфейсом со статикой и автоподнятием. VLAN-интерфейс enp0s3.100 наследует от него провод, но добавляет метку 100.

echo -e "TYPE=eth\nBOOTPROTO=static\nONBOOT=yes" > enp0s3/options

echo -e "TYPE=vlan\nHOST=enp0s3\nVID=100\nBOOTPROTO=static\nONBOOT=yes" > enp0s3.100/options

Теперь прописываем IP-адрес из серверной подсети (помните, на HQ-RTR мы выделили 192.168.10.0/29), шлюз по умолчанию и временный DNS. Почему временный? Потому что наш dnsmasq ещё не запущен, а системе уже сейчас хочется обновляться и резолвить.

echo 192.168.10.2/29 > enp0s3.100/ipv4address

echo default via 192.168.10.1 > enp0s3.100/ipv4route

echo nameserver 8.8.8.8 > enp0s3.100/resolv.conf

systemctl restart network

Перезапуск сети применит адрес и маршрут. Можно представить, как пакет от сервера теперь идет через шлюз .1 и попадает в NAT на роутере.

Следом – сердце сервера, dnsmasq. Этот малыш способен кэшировать запросы, раздавать DHCP (хотя здесь это не нужно) и, главное, служить легковесным DNS-сервером. Он будет знать все внутренние имена и пересылать остальные запросы на 8.8.8.8. Ставим его и редактор nano, если вдруг ещё нет.

apt-get update

apt-get install -y nano dnsmasq

Конфигурационный файл dnsmasq — это мощный, но дружелюбный инструмент. Открываем и наполняем.

nano /etc/dnsmasq.conf

Добавляем следующие строки (адреса оставляем как в примере, они строго соответствуют топологии):

listen-address=192.168.10.2

server=8.8.8.8

address=/hq.example.com/10.10.1.2

ptr-record=2.1.10.10.in-addr.arpa,hq.example.com

address=/br.example.com/10.10.2.2

address=/srv-hq.example.com/192.168.10.2

ptr-record=2.10.168.192.in-addr.arpa,srv-hq.example.com

address=/cli-hq.example.com/192.168.20.2

ptr-record=2.20.168.192.in-addr.arpa,cli-hq.example.com

address=/srv-br.example.com/192.168.3.2

address=/docker.example.com/10.10.1.1

address=/web.example.com/10.10.2.1

Здесь listen-address говорит демону висеть только на нашем серверном IP, server — куда слать незнакомые запросы, а address и ptr-record строят прямые и обратные зоны для всех узлов сети. Обратите внимание, что мы сразу прописали даже будущие сервисы — Docker-хост и веб-сервер. Это даст нам красивый резолвинг без лишней беготни.

Запускаем dnsmasq и прописываем в автозагрузку.

systemctl enable --now dnsmasq

С этого момента любой хост, который использует 192.168.10.2 как DNS-сервер, сможет находить соседей по именам. Красота.

Теперь подготовим административный вход. Создадим специальную учётную запись sshuser. Зачем фиксированный UID 2026? Чтобы при миграциях и бекапах права на файлы не превратились в тыкву, а остались привязаны к этому пользователю.

useradd -u 2026 -m sshuser

passwd sshuser

usermod -aG wheel sshuser

echo "sshuser ALL=(ALL:ALL) NOPASSWD:ALL" >> /etc/sudoers

После этого sshuser сможет делать sudo без пароля — удобно для скриптов и удалённого администрирования, где ввод пароля каждый раз раздражает.

Безопасность SSH — это святое. Сканеры в интернете не дремлют, поэтому мы перенесём демона на порт 2026, оставим доступ только пользователю sshuser, ограничим число попыток аутентификации и повесим предупреждающий баннер. Так bots будут долбиться в закрытую дверь, а мы спокойно заходить на нужный порт.

apt-get install -y openssh-server

nano /etc/openssh/sshd_config

Внутри файла находим и изменяем/добавляем следующие строки:

Port 2026

AllowUsers sshuser

MaxAuthTries 2

Banner /etc/openssh/banner.txt

Сохраняем, выходим. Затем создаём сам баннер — краткое, но строгое предупреждение, которое увидят все подключающиеся.

echo "Authorized access only" > /etc/openssh/banner.txt

systemctl enable --now sshd

Теперь sshd слушает на порту 2026, и только sshuser сможет попытать счастья дважды, прежде чем соединение оборвётся.

Как всегда, не забываем про время. Без корректного часового пояса логи будут врать, а сертификаты могут начать вести себя странно.

apt-get install tzdata -y

timedatectl set-timezone Europe/Moscow

timedatectl status

Всё, сервер готов служить верой и правдой. У нас есть DNS-сервер, знающий всех по именам, и сервер удалённого доступа, который уже не привлекает внимание стандартным портом. Следующая остановка — клиентская машина, где мы проверим, как все эти сервисы работают на практике.
